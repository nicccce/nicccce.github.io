Reinforcement Learning Note

2026-05-01

Chapter 1: Introduction to Reinforcement Learning

1.1 What Is Reinforcement Learning?

Reinforcement Learning (RL) is a mathematical framework for learning-based decision making.

The central idea is simple:

  • An agent interacts with an environment.
  • The agent chooses actions.
  • The environment returns observations and rewards.
  • The agent learns from experience to choose better actions in the future.

In other words, reinforcement learning is about learning how to make decisions and control behavior through trial and error.

Unlike many other machine learning settings, the learner is not just given a fixed dataset. Instead, the learner actively influences what data it will see next.

1.2 The Agent-Environment Loop

A typical RL problem contains the following components:

  • Agent: the learner or decision maker.
  • Environment: the world the agent interacts with.
  • Action: a decision made by the agent.
  • Observation / State: information the agent receives from the environment.
  • Reward: a scalar feedback signal that tells the agent how good or bad the outcome was.

The loop can be summarized as:

  1. The agent observes the current situation.
  2. The agent selects an action.
  3. The environment changes as a consequence of the action.
  4. The agent receives a new observation and a reward.
  5. The agent updates its behavior based on experience.

The goal is to learn a policy that chooses actions to maximize long-term reward.

1.3 How RL Differs from Supervised Learning

In supervised learning, the training data usually consists of input-output pairs:

The learner is told what the correct answer should be.

In reinforcement learning, the situation is different:

The agent is not directly told the correct action. It only receives feedback through rewards.

Important differences include:

  • Data is not i.i.d. Previous actions affect future observations.

  • There is no fixed ground truth label. The agent may only know whether an outcome was good or bad.

  • Actions change the data distribution. The agent's own behavior determines what experience it collects.

  • The goal is long-term performance. A locally good action may lead to poor future outcomes, while a costly action now may produce larger rewards later.

This makes RL a decision-making problem rather than just a prediction problem.

1.4 Examples of RL Problems

RL can be applied whenever an agent must choose actions based on feedback.

Robotics

For a robot:

  • Actions may be motor torques or currents.
  • Observations may be camera images, joint angles, or sensor readings.
  • Rewards may measure task success, speed, stability, or energy efficiency.

Examples include locomotion, navigation, manipulation, and complex physical control tasks.

Games

For a game-playing agent:

  • Actions are legal moves or controller inputs.
  • Observations are board states, pixels, or game states.
  • Rewards may be win/loss signals or game scores.

AlphaGo's famous "Move 37" is an example of RL discovering a strategy that surprised human experts.

Business and Operations

For inventory or purchasing decisions:

  • Actions are what to buy, sell, or allocate.
  • Observations are inventory levels, prices, and demand.
  • Rewards are profit or cost reduction.

Language Models

RL can also be used to improve language models through feedback. In reinforcement learning from human feedback (RLHF), a model is optimized according to preference-based reward signals rather than only next-token prediction.

Image Generation

RL can optimize generative models toward desired objectives, such as human preference, aesthetic quality, or task-specific reward functions.

Chip Design

RL has been used for chip floorplanning, where the agent learns how to place components to optimize performance, power, or area.

1.5 Why Reinforcement Learning Matters

A major reason to study RL is that it combines two powerful ideas:

  • Learning from data
  • Optimization through search or trial and error

Data-driven AI is good at learning patterns from examples. However, if a system only imitates data, it may not discover solutions that are better than the examples it has seen.

RL adds an optimization objective. It allows an agent to search for behavior that achieves a goal, even if that behavior was not explicitly demonstrated by humans.

This is why RL is especially interesting for tasks where:

  • The correct answer is not known in advance.
  • The agent must interact with the world.
  • Success depends on sequences of decisions.
  • New and unexpected strategies may be valuable.

1.6 Learning and Search

The lecture connects reinforcement learning to a broader view of intelligence:

  • Learning uses data to extract patterns.
  • Search uses computation to extract decisions or inferences.
  • Optimization uses computation to improve decisions according to an objective.

Deep reinforcement learning combines these ideas:

  • Deep learning provides scalable function approximation for complex data.
  • Reinforcement learning provides an optimization framework for decision making.

This combination makes it possible to learn policies from high-dimensional observations such as images, language, and sensor streams.

1.7 Why Deep Reinforcement Learning?

Deep reinforcement learning is useful because real-world decision problems often involve rich and complex inputs.

Examples:

  • A robot must interpret camera images and proprioceptive signals.
  • A self-driving car must react to road scenes.
  • A language model must generate useful responses.
  • A game agent may need to act directly from pixels.

Deep networks can represent complex policies and value functions, while RL defines the objective for improving behavior.

Thus:

1.8 Key Takeaways

  • Reinforcement learning is a framework for learning how to act.
  • The agent learns by interacting with an environment and receiving rewards.
  • RL differs from supervised learning because the data depends on the agent's own actions.
  • The goal is not just prediction, but long-term decision quality.
  • RL is useful for games, robotics, operations, language models, image generation, chip design, and many other domains.
  • Deep RL combines the representation power of deep learning with the optimization perspective of reinforcement learning.
  • RL is important because it can discover new solutions rather than only imitate existing data.

Chapter 2: Supervised Learning of Behaviors

2.1 From Supervised Learning to Imitation Learning

Supervised learning estimates a conditional model from labeled examples:

\[ x \to y \]

For behavior learning, the same idea can be used to learn actions from demonstrations:

\[ \text{observation} \to \text{action} \]

This is called imitation learning. Instead of designing a reward function and solving an RL problem, we collect demonstrations from an expert and train a policy to imitate the expert's actions.

The simplest form of imitation learning is behavioral cloning.

2.2 Sequential Decision-Making Setup

In a sequential environment, the agent repeatedly receives observations and chooses actions.

A trajectory can be written as:

\[ \tau = (o_1, a_1, o_2, a_2, \ldots, o_H, a_H) \]

where:

  • o_t is the observation at time t.
  • a_t is the action at time t.
  • H is the horizon, or trajectory length.

In the partially observed case, the policy uses observations:

\[ \pi_\theta(a_t \mid o_t) \]

In the fully observed case, the observation is the state:

\[ o_t = s_t \]

\[ \pi_\theta(a_t \mid s_t) \]

The environment dynamics are usually written as:

\[ p(s_{t+1} \mid s_t, a_t) \]

This expresses the Markov property: the next state depends on the current state and action, not directly on the full past history.

2.3 Demonstration Trajectories

A demonstration trajectory is an example of expert behavior.

For imitation learning, the dataset has the form:

\[ \mathcal{D} = \left\{(o_1^{(i)}, a_1^{(i)}, \ldots, o_H^{(i)}, a_H^{(i)})\right\}_{i=1}^{N} \]

Each trajectory is produced by an expert policy, often denoted pi*.

The goal is to learn a policy pi_theta that chooses actions similar to the expert:

\[ \pi_\theta(a_t \mid o_t) \approx \pi^*(a_t \mid o_t) \]

2.4 Behavioral Cloning

Behavioral cloning treats imitation learning as ordinary supervised learning.

The algorithm is:

  1. Collect demonstrations from an expert.
  2. Create a dataset of observation-action pairs.
  3. Train a policy network by maximum likelihood.

The training objective is:

\[ \arg\max_{\theta} \sum_{i=1}^{N} \sum_{t=1}^{H} \log \pi_\theta(a_t^{(i)} \mid o_t^{(i)}) \]

Equivalently, we minimize the negative log-likelihood of the expert actions.

For discrete actions, such as game controls or language tokens, the model usually outputs logits over actions.

For continuous actions, such as steering angle or robot torque, the model often outputs distribution parameters, such as the mean and variance of a Gaussian.

2.5 Why Behavioral Cloning Is Attractive

Behavioral cloning is simple and scalable.

It can use the same tools as supervised learning:

  • neural network encoders such as ResNet or ViT,
  • cross-entropy loss for discrete actions,
  • Gaussian likelihood or regression loss for continuous actions,
  • large demonstration datasets.

It has been used successfully in systems such as autonomous driving, where the policy maps camera observations to control actions.

However, its simplicity hides an important problem: the data distribution during training is not necessarily the same as the data distribution during deployment.

2.6 Distributional Shift

In supervised learning, we usually assume that training and test examples are drawn from the same distribution.

Behavioral cloning violates this assumption.

During training, observations come from expert trajectories:

\[ o_t \sim p_{\text{data}}(o_t) \]

During deployment, observations come from the learned policy's own behavior:

\[ o_t \sim p_{\pi_\theta}(o_t) \]

If the learned policy makes a small mistake, the environment may move to states that the expert demonstrations rarely visited. The policy is then asked to act on unfamiliar inputs, which can cause more mistakes.

This feedback loop is the key reason behavioral cloning is harder than ordinary supervised learning.

2.7 Error Compounding Over Time

Consider the fully observed case for simplicity.

Define a cost for taking a wrong action:

\[ c(s_t, a_t) = \begin{cases} 0, & a_t = \pi^*(s_t) \\ 1, & \text{otherwise} \end{cases} \]

The expected cost at a state is the probability of making a mistake:

\[ \mathbb{E}_{a_t \sim \pi_\theta(a_t \mid s_t)} \left[c(s_t, a_t)\right] = \pi_\theta(a_t \ne \pi^*(s_t) \mid s_t) \]

The total expected number of mistakes over a horizon H is:

\[ \sum_{t=1}^{H} \mathbb{E}_{s_t \sim p_{\pi_\theta},\, a_t \sim \pi_\theta} \left[c(s_t, a_t)\right] \]

Suppose the learned policy has small error on the training distribution:

\[ \mathbb{E}_{s_t \sim p_{\text{train}}} \left[\pi_\theta(a_t \ne \pi^*(s_t) \mid s_t)\right] \le \epsilon \]

This does not guarantee small rollout error. After the policy makes mistakes, the state distribution can drift away from p_train.

In the worst case, the total error can scale as:

\[ O(\epsilon H^2) \]

This quadratic dependence on the horizon is much worse than the linear behavior we would hope for if each time step were independent:

\[ O(\epsilon H) \]

2.8 Why the Worst Case Can Be Pessimistic

The quadratic bound above is a worst-case analysis.

In many real systems, the environment may be forgiving:

  • small mistakes can be corrected later,
  • nearby states may require similar actions,
  • the policy may generalize beyond the exact training distribution,
  • demonstrations may contain recovery behavior.

This creates a useful paradox: imitation learning can work better when the dataset includes some mistakes and recoveries, because the learner sees how to return from off-expert states.

Still, the theory explains why behavioral cloning is fragile for long-horizon control tasks.

2.9 DAgger: Dataset Aggregation

DAgger, short for Dataset Aggregation, is designed to reduce distributional shift.

The central idea is:

The algorithm is:

  1. Train an initial policy pi_theta on expert demonstrations.
  2. Run pi_theta to collect states or observations from its own rollout distribution.
  3. Ask the expert to label those visited states with the correct actions.
  4. Aggregate the new labeled data into the dataset.
  5. Retrain the policy and repeat.

In notation:

\[ \mathcal{D} \leftarrow \mathcal{D} \cup \mathcal{D}_{\pi} \]

where D_pi contains states visited by the learned policy, labeled by the expert.

DAgger tries to make the training distribution match the deployment distribution:

\[ p_{\text{data}}(o_t) \approx p_{\pi_\theta}(o_t) \]

This directly targets the main failure mode of behavioral cloning.

2.10 Practical Ways to Improve Behavioral Cloning

Besides DAgger, several practical choices can make behavioral cloning more reliable:

  • Use larger and more diverse demonstration datasets.
  • Include recovery demonstrations from imperfect states.
  • Add data augmentation when observations are high-dimensional.
  • Use powerful models that make fewer single-step mistakes.
  • Use multi-task data so the policy learns more robust representations.
  • Combine imitation learning with reinforcement learning when reward feedback is available.

The key is to reduce the mismatch between training inputs and deployment inputs.

2.11 Key Takeaways

  • Behavioral cloning learns a policy by supervised learning on expert demonstrations.
  • The standard objective is maximum likelihood of expert actions.
  • Behavioral cloning is easy to implement but can suffer from distributional shift.
  • Small errors can compound because actions affect future observations.
  • Worst-case imitation error can grow quadratically with the horizon.
  • DAgger reduces distributional shift by collecting expert labels on states visited by the learned policy.
  • Recovery behavior and diverse data are often crucial for long-horizon tasks.

2.12 Vocabulary

Term Meaning
Imitation Learning Learning behavior from expert demonstrations
Behavioral Cloning Supervised learning of a policy from observation-action pairs
Demonstration A trajectory generated by an expert
Expert Policy The policy that provides desired actions, often written pi*
Horizon The length of a trajectory
Maximum Likelihood Training objective that maximizes probability of demonstrated actions
Distributional Shift Mismatch between training distribution and deployment distribution

DAgger Dataset Aggregation, an imitation learning method that labels states visited by the learned policy

异或 DP 的二进分块优化 (Meet in the Middle)

题目来源： https://qoj.ac/problem/15322（2025 ICPC 上海区域赛 I 题）

不难将题目的DP方程化简为 \(dp[i] = \min_{j < i} \{ dp[j] + (a_j \oplus a_i) \}\)

数组长度 \(N \approx 10^5\)，但数值上限 \(M\) 不太大（例如 \(a_i < 2^{18}\)）。

但是不管是往前暴力遍历 \(j\) 还是遍历所有历史值域 \(a_j\) 时间复杂度是 \(O(N^2)\) 或者 \(O(NM)\) ，必然超时。

核心思想

在遍历所有历史值域 \(a_j\) 的基础上对值域进行按位分块。

将 18 位二进制对半劈开：高 9 位、低 9 位。

在查询和更新的时候各分担一半的复杂度

每一半的值域只有 \(2^9 = 512\)。把遍历 \(O(M)\) 转化为遍历 \(O(\sqrt{M})\)。

搭建梯子

常见梯子协议

“梯子”通常指用于翻墙或代理的工具，其核心依赖各种协议来传输数据。下面我列出一些常见的协议，包括传统VPN协议（如OpenVPN、WireGuard）和代理协议（如Shadowsocks、Trojan，常用于绕过审查）。

我将使用表格形式比较，便于阅读。数据基于常见协议的通用特性。

协议名称 基本原理 优点 缺点 Clash 是否支持
OpenVPN 开源协议，使用SSL/TLS加密，建立隧道传输数据。 安全性高，支持强加密（如AES-256）；开源，可自定义配置；兼容性好。 速度较慢（加密开销大）；配置复杂，不适合初学者。 不直接支持，但可以通过Clash Meta的兼容模式或外部配置间接使用。
WireGuard 现代轻量级协议，使用ChaCha20加密，专注于速度和简单性。 速度快、代码简洁、安全性高；易设置，低延迟。 较新协议，可能在某些旧设备上兼容性差；不支持UDP伪装。 是，支持WireGuard协议。
IKEv2/IPSec 结合IKEv2密钥交换和IPSec加密，常用于移动设备。 连接稳定、速度快；适合切换网络（如WiFi到移动数据）。 可能被防火墙检测阻塞；安全性不如OpenVPN。 不支持。
L2TP/IPSec 第2层隧道协议结合IPSec加密，内置于许多系统。 易设置，许多设备原生支持；兼容性广。 速度慢；安全性一般，可能有漏洞。 不支持。
PPTP 旧式点对点隧道协议，使用GRE封装。 速度快、设置简单。 安全性极低，已过时，不推荐使用。 不支持。
Shadowsocks (SS) 代理协议，通过SOCKS5伪装流量，绕过审查。 伪装能力强、速度快；简单高效，常用于翻墙。 加密不如VPN全面；依赖服务器配置。 是，支持SS和SSR（ShadowsocksR）变体。
VMess (V2Ray) V2Ray框架下的协议，支持多路复用和伪装。 高度自定义、绕过检测强；支持多协议 multiplexing。 配置复杂；资源消耗较高。 是，支持V2Ray/VMess。
Trojan 基于HTTPS伪装的代理协议，模拟正常Web流量。 检测难度高、稳定；加密强。 依赖TLS证书；速度中等。 是，支持Trojan。
Hysteria/Hysteria2 基于QUIC的协议，专注于抗丢包和高吞吐。 速度极快、抗干扰强；适合弱网络环境。 配置稍复杂；兼容性取决于客户端。 是，支持Hysteria和Hysteria2。
SOCKS5 通用代理协议，支持UDP/TCP。 灵活、兼容多种应用；速度快。 无加密，安全性低；易被检测。 是，支持SOCKS。

附加说明

对于翻墙需求，优先考虑Shadowsocks、Trojan或Hysteria，这些协议在绕过GFW方面表现更好。传统VPN如WireGuard适合隐私保护，但可能被封锁。

Paper Deep Dive: “Efficient Rectification of Neuro-Symbolic Reasoning Inconsistencies by Abductive Reflection”

TL;DR (Key Takeaways)

This paper introduces Abductive Reflection (ABL-Refl), a novel neuro-symbolic (NeSy) learning framework that efficiently detects and corrects inconsistencies between neural network outputs and symbolic reasoning. Inspired by human dual-process cognition (System 1 & System 2), ABL-Refl uses a reflection mechanism to flag errors in neural predictions and invokes abductive reasoning to fix them.

Why it matters?

  • Efficiency: Avoids costly combinatorial search in prior ABL methods.
  • Accuracy: Outperforms state-of-the-art NeSy methods on Sudoku and combinatorial optimization tasks.
  • Versatility: Works with symbolic, visual, and graph-structured data, leveraging different forms of domain knowledge (logic, math, etc.).

浅谈敏感信息处理

2024-10-19

浅谈敏感信息处理

“信息安全的核心不是完美的技术，而是有效的管理与人性化的考虑。” —— Bruce Schneier

什么是敏感信息？

在项目开源的过程中，不可避免地会碰到敏感信息，例如：

  • 云数据库的账号密码信息
    MySQL等数据库的用户名和密码。
  • 微信登录涉及的 AppID、AppSecret
    这些信息用于第三方服务的 OAuth 认证，如果泄露可能导致账号被恶意使用。
  • API 密钥
    用于调用第三方服务的 API_KEY，如支付网关、地图服务、邮件发送等。
  • SSH 私钥
    服务器管理中的 SSH 私钥文件，如果泄露会让攻击者获得服务器的完全访问权限。
  • JWT Secret
    这些密钥用于生成和验证 JWT，保护用户身份信息。
  • 支付信息
    例如：银行卡号、支付网关的密钥（如支付宝的签名密钥）。
  • 个人身份信息（PII）
    例如：身份证号码、护照信息、电话号码、住址、电子邮件地址等。
  • 密码文件
    例如：包含用户登录密码或哈希密码的配置文件、.env 文件中明文存储的敏感数据。

这些敏感信息如果在开源项目中暴露，可能会引发数据泄露、财务损失、甚至法律责任等严重后果，因此在项目中需要妥善处理和保护。

在 Go 语言里，命名为 main 的包具有特殊的含义。 Go 语言的编译程序会试图把这种名字的包编译为二进制可执行文件。所有用 Go 语言编译的可执行程序都必须有一个名叫 main 的包。一个可执行程序有且仅有一个 main 包。

当编译器发现某个包的名字为 main 时，它一定也会发现名为 main()的函数，否则不会创建可执行文件。 main()函数是程序的入口，所以，如果没有这个函数，程序就没有办法开始执行。

程序编译时，会使用声明 main 包的代码所在的目录的目录名作为二进制可执行文件的文件名。

序列化和反序列化

Java 序列化是一种将对象转换为字节流的过程，以便可以将对象保存到磁盘上，将其传输到网络上，或者将其存储在内存中，以后再进行反序列化，将字节流重新转换为对象。

序列化在 Java 中是通过 java.io.Serializable 接口来实现的，该接口没有任何方法，只是一个标记接口，用于标识类可以被序列化。

当你序列化对象时，你把它包装成一个特殊文件，可以保存、传输或存储。反序列化则是打开这个文件，读取序列化的数据，然后将其还原为对象，以便在程序中使用。

前端基础

2024-04-02

前端基础

1.前端开发介绍

那在讲解web前端开发之前，我们先需要对web前端开发有一个整体的认知。主要明确一下三个问题：

1). 网页有哪些部分组成 ?

文字、图片、音频、视频、超链接、表格等等。

2). 我们看到的网页，背后的本质是什么 ?

程序员写的前端代码 (备注：在前后端分离的开发模式中，)

3). 前端的代码是如何转换成用户眼中的网页的 ?

通过浏览器转化（解析和渲染）成用户看到的网页

浏览器中对代码进行解析和渲染的部分，称为 浏览器内核

而市面上的浏览器非常多，比如：IE、火狐Firefox、苹果safari、欧朋、谷歌Chrome、QQ浏览器、360浏览器等等。 而且我们电脑上安装的浏览器可能都不止一个，有很多。

但是呢，需要大家注意的是，不同的浏览器，内核不同，对于相同的前端代码解析的效果也会存在差异。 那这就会造成一个问题，同一段前端程序，不同浏览器展示出来的效果是不一样的，这个用户体验就很差了。而我们想达到的效果则是，即使用户使用的是不同的浏览器，解析同一段前端代码，最终展示出来的效果都是相同的。

要想达成这样一个目标，我们就需要定义一个统一的标准，然后让各大浏览器厂商都参照这个标准来实现即可。 而这套标准呢，其实早都已经定义好了，那就是我们接下来，要介绍的web标准。

Web标准也称为网页标准，由一系列的标准组成，大部分由W3C（ World Wide Web Consortium，万维网联盟）负责制定。由三个组成部分：

  • HTML: HyperText Markup Language，超文本标记语言。

    • 超文本：超越了文本的限制，比普通文本更强大。除了文字信息，还可以定义图片、音频、视频等内容。

    • 标记语言：由标签构成的语言

      • HTML标签都是预定义好的。例如：使用

        标签展示标题，使用展示超链接，使用展示图片，

      • HTML代码直接在浏览器中运行，HTML标签由浏览器解析。
  • CSS：Cascading Style Sheet，层叠样式表，负责网页的表现（页面元素的外观、位置等页面样式，如：颜色、大小等）。
  • JavaScript：负责网页的行为（交互效果）。

当然了，随着技术的发展，我们为了更加快速的开发，现在也出现了很多前端开发的高级技术。例如：vue、elementui、Axios等等。
